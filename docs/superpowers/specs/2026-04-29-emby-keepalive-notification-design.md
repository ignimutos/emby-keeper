# Emby 保活通知改造设计

## 背景

当前 Emby 通知主要推送调度信息，例如“下一次 Emby 保活将在……进行”。这些消息会在配置重载和重新调度时重复出现，但缺少用户真正关心的执行结果：这次是否成功、保活了哪个视频、Emby 是否记录到了播放、记录成了什么时间、下次何时再跑。

现有结果通知也过于粗糙：

- `embykeeper/emby/main.py` 只汇总“哪些账号成功/失败”
- `embykeeper/emby/api.py` 只记录“共播放了几个视频”
- 通知内容没有以 Emby API 的回写结果为准

## 目标

1. 每个账号每次保活结束后只发一条结果通知。
2. 通知首行直接展示成功/失败结果。
3. 详情字段优先使用 Emby API 回写结果，而不是本地运行布尔值。
4. 不再把“下一次保活将在……”这类调度信息推送到通知通道。
5. 修正 `notifier.immediately` 的配置注释，使其与实际行为一致。

## 非目标

- 不新增新的通知通道或通知类型。
- 不做多账号聚合消息。
- 不实现可折叠/expandable 通知格式。
- 不调整 Telegram 第三方 bot 的推送机制。

## 用户体验

### 成功通知

首行只展示结果摘要：

```text
Emby保活成功｜premises@墨云阁｜片名
```

正文展示详情：

```text
视频ID: abc123
Emby记录时间: 2026-04-29 15:08:12
回写进度: 97% / 1836s
播放次数: 12
下次保活: 2026-05-11 11:18
```

### 失败通知

首行同样直接给出失败结果：

```text
Emby保活失败｜premises@墨云阁｜片名
```

正文格式保持一致，但对取不到的 Emby 字段显示 `未更新`，并增加失败阶段：

```text
视频ID: abc123
Emby记录时间: 未更新
回写进度: 未更新
播放次数: 未更新
失败阶段: 播放后校验未生效
下次保活: 2026-05-11 11:18
```

如果失败发生在拿到目标视频之前，视频名和视频 ID 也显示 `未获取`。

## 数据来源

### 1. 结果判断

通知结果分为两个层次：

- **本地执行结果**：本次保活链路是否跑完，是否拿到了目标视频，是否完成了播放流程
- **Emby 回写结果**：播放前后两次读取 item 时，`UserData` 是否体现出有效更新

为了证明是“这次”保活产生的回写，而不是沿用旧数据，需要在播放前抓取一次基线 item，再在播放后抓取一次最新 item。

通知展示时以 Emby 回写结果优先：

- 如果本地执行成功，且 Emby 回写也成功，通知为 `Emby保活成功`
- 如果本地执行成功，但 Emby 回写未生效或校验失败，通知为 `Emby保活失败`
- 如果本地流程中途失败，通知为 `Emby保活失败`

Emby 回写成功的判定条件定义为以下任一项相对基线发生有效变化：

- `UserData.LastPlayedDate` 更新为更晚时间
- `UserData.PlayCount` 增加
- `UserData.PlaybackPositionTicks` 增加

也就是说，最终通知结果不是简单复用 `watch()` 的布尔值，而是基于播放前后两次拉取的视频详情进行判定。

### 2. Emby 详情字段

保活视频和回写信息以播放后的最新 item 为主，必要时会参考播放前基线 item：

- 视频名：优先取最新 item `Name`，否则回退到基线 item 或本地已知值
- 视频 ID：优先取最新 item `Id`，否则回退到基线 item 或本地已知值
- Emby 记录时间：优先读取最新 item `UserData.LastPlayedDate`；若不存在则显示 `未更新`
- 播放次数：读取最新 item `UserData.PlayCount`；若不存在则显示 `未更新`
- 回写进度秒数：读取最新 item `UserData.PlaybackPositionTicks` 并换算为秒；若不存在则显示 `未更新`
- 回写进度百分比：如果同时存在最新 item 的 `PlaybackPositionTicks` 和 `RunTimeTicks`，则计算百分比；否则显示 `未更新`

“播放时间段”不再尝试构造绝对开始/结束时间，而改为展示更稳定的 `Emby记录时间 + 回写进度`。

### 3. 下次保活时间

下次保活时间不可能来自 Emby API，只能来自本地调度器：

- 独立账号：取对应账号 scheduler 的 `next_time`
- 统一调度账号：取 unified scheduler 的 `next_time`
- 如果当前场景下拿不到调度器时间，则显示 `未计划`

## 触发规则

### 推送

只保留以下两类 Emby 通知：

- 单账号单次保活成功通知
- 单账号单次保活失败通知

每个账号每次保活结束仅推送一条。

### 不推送

以下消息不再进入通知通道：

- `下一次 Emby 保活将在……进行`
- `下一次 Emby 账号 (...) 的保活将在……进行`
- 因配置重载、计划重建、账号增删而重复出现的调度提醒

这些信息仍可保留在本地日志中，但不再使用 `bind(log=True)` 推送。

## 实现方案

### 1. 在 Emby 播放流程中收集结构化结果

当前 `watch()` 只返回布尔值，不足以生成通知。需要把它改为返回结构化结果对象，至少包含：

- `account_spec`
- `success`
- `failure_stage`
- `item_name`
- `item_id`
- `last_played_date`
- `play_count`
- `playback_position_ticks`
- `runtime_ticks`

其中：

- 开始播放前先读取一次 item，保存基线 `UserData`
- 本地播放完成后，再重新读取一次 item 作为最新结果
- 由基线 item 和最新 item 一起判定 Emby 回写是否生效
- 最终通知详情字段以最新 item 为主
- 如果获取 item 失败，则尽可能保留本地已知的 `item_name` / `item_id`

### 2. 在单账号维度发通知

`EmbyManager._watch_main` 目前更偏向批量汇总，需要改成：

- 每个账号完成后立刻生成该账号的结果摘要
- 按单账号发送 `bind(log=True)` 结果通知
- 最后的批量成功/失败日志可以保留给控制台，但不应替代单账号通知

这样统一调度和独立调度都能得到一致体验：一次账号保活，对应一条通知。

### 3. 结果格式化逻辑独立出来

新增一个专门的 Emby 通知格式化函数，用于把结构化结果转换成两段文本：

- 首行摘要
- 详情正文

格式化函数负责：

- 成功/失败标题
- `未更新` / `未获取` / `未计划` 等兜底文案
- 百分比与秒数格式化
- 时间格式化

这样可以避免通知文案散落在 `watch()` 和 `EmbyManager` 中。

### 4. 保留日志通道，取消调度通知推送

调度器的 `on_next_time` 回调仍可写本地日志，但不再进入通知：

- `embykeeper/emby/main.py` 中的调度消息去掉 `bind(log=True)`
- 结果通知继续使用现有 `bind(log=True)` 通道

### 5. 修正配置注释

`notifier.immediately` 的注释需要与实际代码一致：

- `false` 表示走 bot 侧统一日志推送
- `true` 表示走即时消息推送

## 失败阶段建议值

为了让失败通知更可读，失败阶段建议统一为有限集合，例如：

- `登录失败`
- `获取视频失败`
- `开始播放失败`
- `播放中断`
- `播放后校验未生效`
- `结果读取失败`

通知只展示其中一个最贴近用户的阶段值，不直接暴露底层异常堆栈。

## 测试

至少补充围绕通知格式和结果判定的测试，覆盖：

1. Emby 回写成功
   - 有 `LastPlayedDate`
   - 有 `PlayCount`
   - 有 `PlaybackPositionTicks`
   - 输出成功通知

2. 本地播放成功但 Emby 回写未更新
   - `UserData` 没有关键字段
   - 输出失败通知
   - 失败阶段为 `播放后校验未生效`

3. 保活流程在拿到视频前失败
   - 视频名 / 视频 ID 显示 `未获取`
   - 其余 Emby 字段显示 `未更新`

4. 下次保活时间不可用
   - 显示 `未计划`

5. 调度消息不再进入通知通道
   - 调度日志仍可存在
   - 但不触发通知过滤条件

## 风险与取舍

1. 统一调度场景下的“下次保活时间”是共享计划，不是账号独享计划，但这仍比缺失更符合用户预期。
2. 某些 Emby 服务端可能不会稳定回写 `PlaybackPositionTicks`，这时需要接受 `未更新` 结果，而不能伪造本地时间段。
3. 把结果判定切换为“Emby 回写优先”后，可能出现“本地播放成功但通知判失败”的情况；这是刻意行为，因为用户更关心服务端是否真正记账。

## 实施顺序

1. 修正 `notifier.immediately` 注释。
2. 让 Emby 播放流程返回结构化结果，而不是单纯布尔值。
3. 增加 Emby 通知格式化逻辑。
4. 在单账号完成时发送结果通知。
5. 去掉调度消息的通知标记。
6. 补测试并验证成功/失败文案。
